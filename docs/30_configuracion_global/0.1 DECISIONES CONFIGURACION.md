# CATÁLOGO DE DECISIONES CONGELADAS - CONFIGURACIÓN (FASES 31–34)

Este catálogo documenta las decisiones técnicas y de arquitectura para el Centro de Configuración Global, vinculadas a la directiva de no-hardcoding.

---

## Decisiones de Arquitectura de Configuración (D30-01 a D30-06)

### **D30-01: Almacenamiento Centralizado en Base de Datos**
*   **Decisión:** Toda configuración de tenant (colores, logotipos, variables de correo, API keys, números de teléfono) se almacenará en la base de datos (PostgreSQL), nunca en variables de entorno `.env` en el servidor (a excepción de secretos globales del sistema como credenciales maestras de base de datos o claves del servicio Supabase).
*   **Razón:** Posibilitar que cada inquilino configure de manera independiente su entorno (incluyendo pasarelas y tokens) de forma dinámica sin desplegar código ni reiniciar servidores.

### **D30-02: Esquema JSONB con Validación de Esquema**
*   **Decisión:** Las configuraciones se estructurarán en columnas `jsonb` en lugar de crear cientos de columnas específicas para cada configuración, lo que previene la rigidez del esquema. Se utilizarán triggers de PostgreSQL para validar el formato de las configuraciones contra un esquema JSON definido.
*   **Razón:** Facilidad de extensión sin migraciones físicas de DDL constantes cuando se añadan nuevos campos de configuración.

### **D30-03: Cero Hardcoding de Teléfonos e Identidades**
*   **Decisión:** Queda estrictamente prohibido escribir números de teléfono de contacto, nombres de empresa, URLs de soporte, correos o branding directamente en el código de frontend o backend (ej: `+573001112233` o `resend.send("soporte@empresa.com")`).
*   **Razón:** Cumplimiento estricto del protocolo `0.4 CONFIGURACION_GLOBAL_OBLIGATORIA.md`. La violación de esta regla causa fallo automático de la fase.

### **D30-04: Estructura de Campos Personalizados (Custom Fields)**
*   **Decisión:** Se añadirá una columna `custom_fields jsonb DEFAULT '{}'` a las tablas principales del ERP (`clients`, `requirements`, `quotes`, `jobs`, `invoices`, `warranties`, `users`). Una tabla de control `custom_field_definitions` definirá los campos dinámicos válidos por tenant y tipo de entidad, y triggers validarán que el payload JSON de `custom_fields` cumpla con los tipos y formatos permitidos.
*   **Razón:** Permitir a los inquilinos añadir metadatos dinámicos sin modificar físicamente la base de datos ni romper las políticas de RLS.

### **D30-05: Enrutamiento Dinámico de Notificaciones**
*   **Decisión:** Las notificaciones despachadas por el bus de eventos resolverán dinámicamente los destinatarios y canales (Telegram, Email, WhatsApp, SMS, Push) leyendo la configuración de enrutamiento y preferencias del tenant almacenadas en base de datos.
*   **Razón:** Separar la lógica operativa de las reglas de negocio, permitiendo reconfigurar quién recibe alertas sin modificar los triggers operacionales.

### **D30-06: Aislamiento RLS en Configuraciones**
*   **Decisión:** RLS estará habilitado en todas las tablas de configuración. Un usuario normal del tenant solo puede leer las configuraciones. Solo los roles `ADMINISTRADOR_TENANT` o `GERENTE` tienen permisos de actualización. Super Admins pueden gestionar configuraciones globales.
*   **Razón:** Garantizar que los inquilinos no puedan acceder a configuraciones de terceros y que se mantenga el aislamiento estricto de SaaS.
